Usage

  • Click a module node to open the detail panel on the right
  • Use the search box to filter by name, type, path, or alarm condition
  • Filter by severity or unresolved reason using the dropdowns
  • Hover over alarm nodes for full details in a tooltip
  • In the detail panel, switch tabs to view alarms, state machines, IO, etc.
  • Click a state in the SVG diagram to show its Structured Text code

ISA-88 Hierarchy

The tree follows the ISA-88 physical model:

UM — Unit Module (top-level machine)
EM — Equipment Module (stations/subsystems)
CM — Control Module (actuators/sensors)
DM — Device Module (individual I/O)

Each layer owns its own alarms. Alarms propagate upward via _AlarmsPresentXxx flags from child modules to their parents.

Alarm Severities

Critical — Machine stop, operator intervention required
Process — Process deviation, may self-recover
Advisory — Non-critical warning, logged for review
Information — Status notification, no action needed

DM_TriggeredLatch Alarms

Each alarm is an instance of DM_TriggeredLatch. When the trigger condition becomes TRUE, the alarm latches active and remains until explicitly acknowledged by the operator (or auto-acknowledged if configured).

The ._Latched variable is the live alarm state. The analyser reads the trigger condition and delay from the code body to populate the tree.

Severity Determination

Severity is determined by which _AlarmsPresentXxx method the alarm is raised in:

  • _AlarmsPresentCritical → Critical
  • _AlarmsPresentProcess → Process
  • _AlarmsPresentAdvisory → Advisory
  • _AlarmsPresentInformation → Information

If the alarm is not found in any of these methods, it is marked Unresolved.

Unresolved Reasons

BASECLASS Defined in base class, not overridden
NO_METHOD Module has no _AlarmsPresent method
MISSING Alarm not referenced in any severity method
DEAD Alarm declared but never triggered in code

State Machines

State machines use DM_StateMachine with a CASE on an ENUM type. Transitions are performed via GotoState(). The analyser extracts all states, their code bodies, and transitions between them.

In the SVG diagram: green = initial state, red = error/timeout state, grey = transition state. Click any state to view its Structured Text code.

Zenon HMI Integration

Alarm variables are mapped to Zenon via the variablePath field (e.g. S0_Objects.Machine.ALM_xxx._Latched). The alarm list export can generate a Zenon-compatible CSV for import into the HMI alarm system.
